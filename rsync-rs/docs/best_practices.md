# Лучшие практики реализации

## Архитектура

Разделить код на четыре слоя:

1. Scanner

   Обходит исходную и целевую директории, собирает метаданные файлов.

2. Planner

   Сравнивает состояния и строит список операций: copy, update, delete, mkdir, skip.

3. Executor

   Выполняет план. В dry-run режиме не меняет файловую систему.

4. Reporter

   Отвечает за table/json вывод, прогресс, статистику и логи.

Такой дизайн легко тестировать: самые важные проверки находятся в planner, где не нужно трогать реальную файловую систему.

## Безопасность данных

- По умолчанию не удалять файлы из target.
- Для `--delete` показывать количество удаляемых файлов.
- Поддержать `--dry-run` для всех опасных операций.
- Большие файлы писать во временный файл, потом делать atomic rename.
- Не перезаписывать файл, если копирование не завершилось.
- Для `--trash` перемещать удаляемые файлы в карантин.

## Производительность

- Сначала использовать quick check: размер и modified time.
- Хешировать файл только при `--checksum` или если quick check недостаточен.
- Для больших файлов читать буферами.
- Не держать в памяти содержимое файлов целиком.
- Ограничить параллелизм, чтобы не перегружать storage на телефоне.
- Отдельно тестировать сценарии: много маленьких файлов и несколько больших файлов.

## CLI UX

- Команды должны быть предсказуемыми:

  ```bash
  rsync-rs plan ./src ./dst
  rsync-rs sync ./src ./dst --dry-run
  rsync-rs check ./src ./dst
  ```

- Ошибки писать понятно: какой файл, какая операция, какая причина.
- Для скриптов нужен стабильный JSON output.
- Exit codes:
  - `0`: успех;
  - `1`: частичная ошибка;
  - `2`: ошибка аргументов или конфигурации;
  - `3`: проверка `check` нашла расхождения.

## Тестирование

- Unit-тесты для planner.
- Integration-тесты через временные директории.
- Property-based тесты для правил include/exclude.
- Snapshot-тесты для CLI output.
- Тесты на Unicode paths и пробелы в именах файлов.
- Тесты на конфликт: файл в source, директория с тем же именем в target.
