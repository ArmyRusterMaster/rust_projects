# JWT And Tokens

Токены являются переносимыми доказательствами, поэтому их потеря часто равна компрометации доступа. Проектировать токены нужно как высокорисковую часть системы.

## Типы токенов

### Access Token

Короткоживущий токен для доступа к API.

Рекомендации:

- Lifetime: обычно 5-15 минут.
- Проверять `iss`, `aud`, `exp`, `nbf`, `iat`.
- Использовать `scope` или `permissions` только как часть общей модели авторизации.
- Не хранить чувствительные данные в payload.

### Refresh Token

Долгоживущий токен для получения новых access tokens.

Рекомендации:

- Хранить только хеш refresh token.
- Использовать rotation: каждый refresh выдает новый refresh token.
- Обнаруживать token reuse и отзывать всю token family.
- Привязывать token family к user, client, device и session.

### ID Token

OIDC-токен, который подтверждает аутентификацию пользователя для клиента. Не использовать как bearer token для API.

## JWT

JWT удобен для stateless-проверки, но сложнее в отзыве.

Используйте JWT, если:

- Нужно много независимых resource servers.
- Нужна проверка без запроса в auth service.
- Есть понятный key rotation и JWKS.

Не используйте JWT как сессию по умолчанию, если:

- Нужен мгновенный logout.
- Нужно часто отзывать права.
- Claims быстро меняются.

## Алгоритмы подписи

Предпочтительно:

- EdDSA, если вся инфраструктура поддерживает Ed25519.
- ES256, если нужна широкая совместимость.
- RS256, если есть legacy-интеграции.

Избегать:

- `none`.
- Слабых shared secrets.
- Смешивания symmetric и asymmetric algorithms для одного issuer без строгой проверки.

## JWKS

JWKS endpoint публикует публичные ключи для проверки JWT.

Практики:

- Каждый ключ имеет `kid`.
- Старый ключ остается опубликованным до истечения всех токенов, подписанных им.
- Новый ключ публикуется до начала активной подписи.
- Resource servers кешируют JWKS с разумным TTL.

## Opaque Tokens

Opaque token не содержит claims и проверяется через introspection endpoint или внутреннее хранилище.

Плюсы:

- Мгновенный отзыв проще.
- Меньше риск утечки данных в payload.
- Можно централизованно применять политику.

Минусы:

- Resource server зависит от auth service или кеша.
- Нужно проектировать latency и отказоустойчивость.
